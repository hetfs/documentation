.vale.ini for Technical Writing

This .vale.ini configuration is designed to enhance technical documentation quality by enforcing clarity, consistency, and professionalism. It integrates base styles, supplementary styles, and format-specific rules to automate documentation linting and ensure adherence to best practices.

📝 .vale.ini Configuration​

# Vale Configuration for Technical Writing

# Define the path to custom styles
StylesPath = styles

# Set the minimum alert level (suggestion, warning, or error)
MinAlertLevel = suggestion

# Automatically install external style guides (optional)
Packages = Microsoft, write-good, alex, Hugo, MDX

# Ignore specific files and directories
ignore = node_modules, vendor, dist

# File-specific configurations for Markdown, MDX, and reStructuredText
[*.{md,mdx,rst}]
BasedOnStyles = Vale, Microsoft, write-good, alex

# Enforce best practices for technical writing
write-good.Passive = yes  # Detect passive voice
write-good.Weak = yes  # Flag weak words
write-good.Complex = yes  # Highlight complex phrases

# Ensure inclusive language
alex.All = yes

# Custom Vale rules for consistency
Vale.Terms = yes  # Enforce terminology consistency
Vale.Spelling = yes  # Enable spell checking
Vale.Grammar = yes  # Check grammar errors

# Ignore specific syntax for Hugo and MDX
[*.{md,mdx}]
IgnorePatterns = (?s)\{\{.*?\}\}, (?s)---.*?---

[*.mdx]
IgnorePatterns = (?s)<.*?>

🔹 Key Features​

  1. Base & Supplementary Styles:

    • Uses Microsoft as the primary writing style.
    • Supplements with write-good (readability) and alex (inclusive language).

  2. Custom Rules:

    • Flags passive voice, weak words, and complex phrases.
    • Enforces terminology, spelling, and grammar checks.

  3. Format-Specific Handling:

    • Supports Markdown (.md), MDX (.mdx), and reStructuredText (.rst).
    • Ignores Hugo shortcodes and MDX JSX components.

  4. Directory & File Exclusions:

    • Prevents linting of unnecessary files like node_modules and vendor.

  5. Flexible Alert Levels:

    • Default set to suggestion to flag minor issues proactively.

🚀 How to Use This Configuration​

1️⃣ Install Vale​

Ensure Vale is installed on your system:

brew install vale  # macOS (Homebrew)
choco install vale  # Windows (Chocolatey)
scoop install vale  # Windows (Scoop)

Alternatively, install manually from Vale's GitHub Releases.

2️⃣ Add .vale.ini to Your Project​

Save the .vale.ini file in your repository root.

3️⃣ Sync Required Styles​

Download and initialize the styles:

vale sync

4️⃣ Run Vale to Lint Documentation​

Use Vale to check a file:

vale docs/example.md

Or check the entire documentation:

vale .

🔄 CI/CD Integration (GitHub Actions)​

To automate documentation linting, integrate Vale into CI/CD workflows.

Create a .github/workflows directory and add a vale-linter.yml file in it.

The official GitHub Action for Vale

📌 GitHub Actions Workflow​

name: Documentation Linting

on:
  push: # Run on every push to the main branch
    branches:
      - main
  pull_request: # Run on pull requests targeting any branch
    branches:
      - "*"

jobs:
  lint:
    name: Run Vale Linting
    runs-on: ubuntu-latest

    steps:
      - name: Checkout Repository
        uses: actions/checkout@v4

      # Install external parsers if needed
      # For AsciiDoc users:
      - name: Install Asciidoctor
        run: sudo apt-get install -y asciidoctor

      # For reStructuredText users:
      - name: Install docutils
        run: sudo apt-get install -y docutils

      - name: Run Vale Linter
        uses: errata-ai/vale-action@v2.1.1
        with:
          version: 2.17.0
          files: .
          reporter: github-pr-check # github-pr-check, github-pr-review, github-check
          fail_on_error: true
          filter_mode: nofilter # added, diff_context, file, nofilter
          vale_flags: "--glob=*.md"
          # token: ${{secrets.VALE_GITHUB_TOKEN}}

🔹 CI/CD Features​

✅ Triggers on Push & Pull Requests: Runs automatically on every push to main and pull requests.
✅ Flags Documentation Issues in PR Checks: Highlights issues before merging.
✅ Supports External Parsers: Installs Asciidoctor and docutils for additional formats.
✅ Fails PR if Critical Errors are Detected (fail_on_error: true).

git add .
git commit -m “changes”
git push

Go to the Actions tab on your repository. You should see the workflow running:

🔧 Customization Options​

  • Lint specific directories (e.g., docs/ only):

    with:
      files: docs/

  • Use PR review comments instead of PR checks:

    with:
      reporter: github-pr-review

  • Exclude files using glob patterns:

    with:
      vale_flags: "--glob=!docs/excluded.md"

📂 Recommended Repository Structure​

.github/
├── styles/
│   └── vocab.txt  # Custom terminology rules
└── workflows/
    └── vale-linter.yml  # GitHub Actions workflow
.vale.ini  # Vale configuration file
docs/
└── example.md  # Documentation file

🛠️ Troubleshooting Vale Issues​

❌ 1️⃣ Missing Styles?​

Run:

vale sync

Verify that styles/ contains required styles (Microsoft, write-good, etc.).

❌ 2️⃣ .vale.ini Parsing Errors?​

Ensure correct syntax:

  • No missing commas
  • Proper formatting of BasedOnStyles
  • Valid IgnorePatterns

❌ 3️⃣ Excluding Unwanted Files?​

Modify:

ignore = node_modules, vendor, dist, .git, .github

This prevents linting unnecessary directories.
